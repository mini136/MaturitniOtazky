19 – DNS – Funkce, hierarchie, konfigurace

Co je DNS

Domain Name System – hierarchický distribuovaný systém pro překlad doménových jmen na IP adresy (a naopak). Funguje na portu 53 (UDP pro dotazy, TCP pro zónové přenosy a velké odpovědi).

Hierarchie překladu – od lokálního po globální

1. Lokální soubor hosts

# Windows: C:\Windows\System32\drivers\etc\hosts
# Linux:   /etc/hosts

127.0.0.1       localhost
192.168.1.10    server.firma.local    server
10.0.0.5        db.firma.local

Kontrolován před DNS. Ruční mapování, vhodné pro malý počet záznamů.

2. DNS cache (lokální)

OS si pamatuje nedávné DNS odpovědi dle TTL (Time to Live).

# Windows
ipconfig /displaydns          # zobrazit cache
ipconfig /flushdns            # vymazat cache

# Linux
systemd-resolve --statistics  # statistiky
systemd-resolve --flush-caches

3. Rekurzivní DNS resolver

Server, který provádí celý překlad za klienta. Typicky DNS server ISP nebo veřejný (8.8.8.8, 1.1.1.1).

4. Root DNS servery

  • 13 logických serverů (A–M root), distribuovaných pomocí anycast
  • Odkazují na TLD servery

5. TLD (Top-Level Domain) servery

  • Správa domén nejvyšší úrovně: .com, .cz, .org, .net
  • Odkazují na autoritativní servery konkrétní domény

6. Autoritativní DNS server

Obsahuje skutečné záznamy pro doménu. Poskytuje konečnou odpověď.

Proces DNS překladu

Uživatel zadá: www.example.com

1. Klient → lokální hosts soubor         (nenalezeno)
2. Klient → DNS cache                    (nenalezeno)
3. Klient → Rekurzivní resolver (8.8.8.8)
4.          Resolver → Root server        "Kdo spravuje .com?"
5.          Resolver ← Root server        "Zeptej se .com TLD na 192.5.6.30"
6.          Resolver → TLD server (.com)  "Kdo spravuje example.com?"
7.          Resolver ← TLD server         "Zeptej se ns1.example.com (93.184.216.34)"
8.          Resolver → Autoritativní NS    "Jaká je IP www.example.com?"
9.          Resolver ← Autoritativní NS    "A záznam: 93.184.216.34"
10. Klient ← Resolver                     "IP je 93.184.216.34" (+ uloží do cache)

Iterativní vs. Rekurzivní dotaz

TypPopis
RekurzivníKlient → resolver: „Zjisti to za mě celé"
IterativníResolver → server: „Nevím, ale zeptej se tam →" (referral)

Konfigurace DNS serveru (BIND na Linuxu)

named.conf

// /etc/bind/named.conf.local

zone "firma.local" {
    type master;
    file "/etc/bind/db.firma.local";
};

zone "1.168.192.in-addr.arpa" {          // reverzní zóna
    type master;
    file "/etc/bind/db.192.168.1";
};

Zónový soubor

; /etc/bind/db.firma.local
$TTL 86400
@   IN  SOA ns1.firma.local. admin.firma.local. (
            2024010101  ; Serial
            3600        ; Refresh (1h)
            900         ; Retry (15min)
            604800      ; Expire (7 dní)
            86400       ; Minimum TTL (1 den)
        )

; Name servery
@       IN  NS      ns1.firma.local.
@       IN  NS      ns2.firma.local.

; A záznamy
ns1     IN  A       192.168.1.2
ns2     IN  A       192.168.1.3
www     IN  A       192.168.1.10
mail    IN  A       192.168.1.20
ftp     IN  CNAME   www.firma.local.

; MX záznam
@       IN  MX  10  mail.firma.local.

DNS nástroje

# nslookup – základní dotazy
nslookup www.google.com
nslookup -type=MX google.com
nslookup www.google.com 8.8.8.8         # dotaz na konkrétní server

# dig – detailní dotazy (Linux)
dig www.google.com
dig @8.8.8.8 www.google.com A
dig google.com MX +short
dig -x 93.184.216.34                    # reverzní lookup

# host (Linux)
host www.google.com
host -t MX google.com

DNSSEC

Podepisuje DNS záznamy digitálním podpisem → ověřitelná autenticita a integrita odpovědí. Neposkytuje šifrování (to řeší DoH / DoT).

DoH (DNS over HTTPS) na portu 443 a DoT (DNS over TLS) na portu 853 šifrují DNS dotazy – ochrana soukromí.
