05 – Architectural design patterns — MVC, Multitier, Monolithic, P2P, Client/Server

🏗️ ČÁST 1 — Architektonické vzory

MVC / MVP / MVVM

VzorRole komponentTok komunikaceVýhody / nevýhodyTypické použití
MVC Model = data + business logika, View = zobrazení, Controller = reakce na vstupy uživatele Uživatel pracuje s View → Controller mění Model → View se překreslí podle stavu Modelu + Jasné oddělení odpovědností, dobré pro web; − U větších aplikací mohou kontrolery příliš narůstat Server-side weby (Django, Rails, ASP.NET MVC)
MVP Presenter obsahuje většinu prezentační logiky, View je „pasivní“ View předá akci Presenteru → Presenter načte/změní Model → Presenter řekne View, co má zobrazit + Snadnější testování prezentační logiky; − Více „lepidla“ mezi vrstvami (větší množství kódu) Desktop/mobile UI (WinForms, starší Android)
MVVM ViewModel připravuje data pro UI, View je napojené přes data binding View je svázané s ViewModelem (binding) → změna dat se promítá automaticky oběma směry + Méně ručního propojování UI a logiky, reaktivní přístup; − Složitější debug bindingu WPF, moderní frontend frameworky (Angular, Vue konceptuálně)
  • Kdy použít MVC: když chceš jednoduché a přehledné oddělení dat, logiky a prezentace.
  • Kdy použít MVP: když chceš maximalizovat testovatelnost prezentační vrstvy.
  • Kdy použít MVVM: když framework podporuje binding a aplikace je hodně UI-orientovaná.

Vícevrstvá architektura (Multitier / N-tier)

  • Presentation Layer — UI, validace vstupů, práce s uživatelem.
  • Application/Business Layer — aplikační pravidla, use-cases, orchestruje tok operací.
  • Data Access Layer — komunikace s databází, ORM/repozitáře, transakce.
  • Data Layer — databáze, souborová úložiště, externí systémy.

Typický tok požadavku: UI přijme požadavek → business vrstva rozhodne, co se má stát → data access vrstva provede čtení/zápis → výsledek se vrací zpět uživateli.

Hlavní přínos: oddělení odpovědností, lepší údržba a testování. Riziko: příliš mnoho vrstev může přinést zbytečnou režii a složitost.

🌐 ČÁST 2 — Komunikace klient-server

Klient-Server, P2P

  • Klient-Server — klient posílá požadavky, server poskytuje služby/data. Výhoda je centralizovaná správa, autentizace, zálohování. Nevýhoda je závislost na dostupnosti serveru (single point of failure bez redundance).
  • P2P (Peer-to-Peer) — každý uzel je zároveň klient i server. Výhoda je decentralizace a škálování přidáváním uzlů. Nevýhoda je složitější bezpečnost, důvěra mezi uzly a koordinace.
  • Hybridní model — kombinace obou přístupů (např. centrální registr + přímá P2P výměna dat).

Příklady: klient-server (webové aplikace, e-mail, firemní IS), P2P (BitTorrent, blockchain sítě, některé VoIP systémy).

Ukázkový kód — MVC (Python)

Model — datová vrstva spravuje seznam studentů. Obsahuje logiku pro přidání, odebrání a získání dat.
# === MODEL ===
class StudentModel:
    """Datova vrstva - spravuje data"""
    def __init__(self):
        self._studenti = []

    def pridej(self, jmeno, vek):
        student = {"id": len(self._studenti) + 1, "jmeno": jmeno, "vek": vek}
        self._studenti.append(student)
        return student

    def vsichni(self):
        return list(self._studenti)

    def odeber(self, id):
        self._studenti = [s for s in self._studenti if s["id"] != id]
View — prezentační vrstva. Pouze zobrazuje data, nemá logiku. Controller — zprostředkovává mezi modelem a view.
# === VIEW ===
class StudentView:
    """Prezentacni vrstva - zobrazuje data"""
    def zobraz_studenty(self, studenti):
        print("  Seznam studentu:")
        for s in studenti:
            print(f"    [{s['id']}] {s['jmeno']}, {s['vek']} let")

    def zobraz_zpravu(self, zprava):
        print(f"  > {zprava}")

# === CONTROLLER ===
class StudentController:
    """Ridici vrstva - zprostredkovava Model a View"""
    def __init__(self, model, view):
        self.model = model
        self.view = view

    def pridej_studenta(self, jmeno, vek):
        student = self.model.pridej(jmeno, vek)
        self.view.zobraz_zpravu(f"Pridan: {student['jmeno']}")

    def zobraz_vsechny(self):
        studenti = self.model.vsichni()
        self.view.zobraz_studenty(studenti)

# Pouziti MVC
model = StudentModel()
view = StudentView()
controller = StudentController(model, view)
controller.pridej_studenta("Alice", 20)
controller.pridej_studenta("Bob", 22)
controller.zobraz_vsechny()

Ukázkový kód — TCP klient-server

TCP server s vlákny — každé připojení obsluhuje samostatné vlákno. Přijímá data a odesílá je zpět velkými písmeny (echo server).
import socket
import threading

def tcp_server():
    """Jednoduchy TCP echo server s vlakny"""
    server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server.bind(('localhost', 65432))
    server.listen(5)
    print("  Server posloucha na localhost:65432")

    def obsluha_klienta(conn, addr):
        print(f"  Pripojeni od {addr}")
        while True:
            data = conn.recv(1024)
            if not data:
                break
            odpoved = data.decode().upper()
            conn.sendall(odpoved.encode())
        conn.close()
        print(f"  Odpojeni {addr}")

    while True:
        conn, addr = server.accept()
        vlakno = threading.Thread(target=obsluha_klienta, args=(conn, addr))
        vlakno.daemon = True
        vlakno.start()
TCP klient — připojí se k serveru, odešle zprávu, přijme odpověď a uzavře spojení.
def tcp_klient():
    """TCP klient"""
    klient = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    klient.connect(('localhost', 65432))
    klient.sendall(b"Ahoj servere!")
    odpoved = klient.recv(1024)
    print(f"  Odpoved: {odpoved.decode()}")
    klient.close()
