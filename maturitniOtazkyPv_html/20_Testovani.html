🧪 ČÁST 1 — Úrovně testování (pyramida)

Pyramida testování

Testovací pyramida říká, jak velké zastoupení by měl mít každý typ testu. Dole základna = nejvíc testů, nahoře špička = nejméně testů.

ÚroveňCo testujeRychlostPočetPříklad nástroje
Unit testyJednotlivé funkce/metody izolovaněVelmi rychlé (ms)Nejvíc (~70 %)unittest, pytest
Integrační testySpolupráce více komponent (DB, API)Střední (sekundy)Méně (~20 %)pytest + TestClient
E2E testyCelý systém z pohledu uživatelePomalé (minuty)Málo (~10 %)Selenium, Playwright
Proč pyramida a ne ledovec? Čím výš, tím dražší na údržbu, pomalejší a křehčí. Unit testy jsou nejrychlejší zpětná vazba pro vývojáře.

Další typy testů

TypPopis
Smoke testZákladní sanity check po deployi — "web vůbec běží?"
Regresní testOvěření, že opravená chyba se nevrátila (a stará funkce stále funguje)
Zátěžový (load) testChování pod velkým počtem uživatelů/požadavků
Bezpečnostní testPenetrační testování, SQL injection, XSS, …
Akceptační test (UAT)Zákazník ověří, že produkt splňuje požadavky

White-box vs Black-box testování

📝 ČÁST 2 — Principy testování (AAA, FIRST)

AAA pattern

Každý unit test by měl být rozdělen do tří jasných fází:

  1. Arrange — příprava vstupů, objektů a prostředí (setUp)
  2. Act — provedení testované operace (jedno volání)
  3. Assert — ověření výsledku pomocí assert metod
Každý test by měl testovat jednu věc. Pokud test selže, musí být ihned jasné co a proč.

FIRST princip

PísmenoAnglickyCo znamená
FFastTesty běží rychle — feedback do sekund
IIndependentTesty nezávisí na sobě, lze spustit v libovolném pořadí
RRepeatableVždy stejný výsledek — žádná závislost na čase, síti, náhodě
SSelf-validatingAutomaticky vyhodnotí pass/fail — žádné ruční porovnávání
TTimelyPíší se včas — ideálně před kódem (TDD)
🔄 ČÁST 3 — TDD, Ukázkový kód a Test Doubles

TDD cyklus (Red–Green–Refactor)

Test-Driven Development = nejprve napsán test, pak teprve implementace. Cyklus se opakuje pro každou novou funkci.

  1. RED — napiš test → selže (funkce ještě neexistuje)
  2. GREEN — napiš minimální kód → test projde
  3. REFACTOR — vylepši kód, odstraň duplikace (testy stále zelené)
Výhody TDD: Nutí přemýšlet o rozhraní před implementací. Každý řádek kódu má svůj test. Refaktoring je bezpečný.

Test Doubles (náhradní objekty)

Když testovaná třída závisí na databázi, síti nebo jiné komponentě, používáme náhradní objekty pro izolaci:

TypPopisPříklad
DummyObjekt, který se předá, ale nikdy se nepoužijeNull objekt v parametru
StubVrací předpřipravené odpovědi, nereaguje na jiné voláníreturn_value = 42
MockOvěřuje, že byly volány správné metody se správnými argumentyassert_called_once_with(1)
SpySkutečný objekt, který si zaznamenává voláníSledování bez nahrazení
FakeFunkční, ale zjednodušená implementace (např. in-memory DB)SQLite místo PostgreSQL

Ukázkový kód — Python

Třída Kalkulacka — testovaná třída s metodami pro sčítání, odčítání, násobení a dělení (včetně ošetření dělení nulou).
class Kalkulacka:
    """Jednoducha kalkulacka pro demonstraci testovani"""

    def secti(self, a, b):
        return a + b

    def odecti(self, a, b):
        return a - b

    def vynasob(self, a, b):
        return a * b

    def vydel(self, a, b):
        if b == 0:
            raise ZeroDivisionError("Nelze delit nulou!")
        return a / b

    def prumer(self, cisla):
        if not cisla:
            raise ValueError("Prazdny seznam!")
        return sum(cisla) / len(cisla)
Unit testy s unittest — testovací třída TestKalkulacka testuje všechny metody kalkulačky. setUp() inicializuje kalkulačku před každým testem. assertRaises ověřuje výjimky.
import unittest

class TestKalkulacka(unittest.TestCase):
    """Testy pro Kalkulacku"""

    def setUp(self):
        """Arrange - pred kazdym testem"""
        self.calc = Kalkulacka()

    def test_secti_kladna(self):
        # Arrange (hotovo v setUp)
        # Act
        vysledek = self.calc.secti(2, 3)
        # Assert
        self.assertEqual(vysledek, 5)

    def test_secti_zaporna(self):
        self.assertEqual(self.calc.secti(-1, -1), -2)

    def test_secti_nula(self):
        self.assertEqual(self.calc.secti(0, 0), 0)

    def test_odecti(self):
        self.assertEqual(self.calc.odecti(10, 3), 7)

    def test_vynasob(self):
        self.assertEqual(self.calc.vynasob(4, 5), 20)

    def test_vydel(self):
        self.assertAlmostEqual(self.calc.vydel(10, 3), 3.333, places=2)

    def test_vydel_nulou(self):
        """Test vyjimky pri deleni nulou"""
        with self.assertRaises(ZeroDivisionError):
            self.calc.vydel(10, 0)

    def test_prumer(self):
        self.assertEqual(self.calc.prumer([1, 2, 3, 4, 5]), 3.0)

    def test_prumer_prazdny(self):
        with self.assertRaises(ValueError):
            self.calc.prumer([])
Parametrizované testy — subTest() umožňuje testovat více vstupů v jednom testu. Při selhání jednoho případu se zbylé stále provedou.
class TestParametrizovane(unittest.TestCase):
    """Parametrizovane testy s subTest"""

    def test_secti_parametry(self):
        calc = Kalkulacka()
        test_cases = [
            (2, 3, 5),
            (-1, 1, 0),
            (0, 0, 0),
            (100, -50, 50),
            (0.1, 0.2, 0.3),
        ]
        for a, b, ocekavano in test_cases:
            with self.subTest(a=a, b=b):
                self.assertAlmostEqual(calc.secti(a, b), ocekavano, places=5)
Mocking — unittest.mock.Mock nahrazuje reálnou databázi falešnou. Testujeme StudentManager izolovaně od databáze pomocí mock.return_value.
from unittest.mock import Mock, patch

class Databaze:
    def najdi_studenta(self, id): pass
    def uloz_studenta(self, student): pass

class StudentManager:
    def __init__(self, db):
        self.db = db

    def get_student_info(self, id):
        student = self.db.najdi_studenta(id)
        if student is None:
            raise ValueError(f"Student {id} nenalezen")
        return f"{student['jmeno']} (vek: {student['vek']})"

class TestStudentManagerMock(unittest.TestCase):
    def test_get_info(self):
        # Arrange - mock databaze
        mock_db = Mock(spec=Databaze)
        mock_db.najdi_studenta.return_value = {"jmeno": "Alice", "vek": 20}

        manager = StudentManager(mock_db)

        # Act
        info = manager.get_student_info(1)

        # Assert
        self.assertEqual(info, "Alice (vek: 20)")
        mock_db.najdi_studenta.assert_called_once_with(1)

    def test_student_nenalezen(self):
        mock_db = Mock(spec=Databaze)
        mock_db.najdi_studenta.return_value = None

        manager = StudentManager(mock_db)

        with self.assertRaises(ValueError):
            manager.get_student_info(999)
📊 ČÁST 4 — Code Coverage a pytest

Code Coverage (pokrytí kódem)

Metrika vyjadřující, kolik procent kódu bylo spuštěno při testování.

Typ pokrytíCo měří
Statement coverageKolik řádků/příkazů bylo vykonáno
Branch coverageKolik větví (if/else, switch) bylo projito — silnější než statement
Path coverageVšechny možné kombinace cest — velmi nákladné
100 % coverage neznamená 100 % správnost. Lze mít plné pokrytí a stále buggy testy. Coverage je nutná podmínka, ne postačující.
Typický cíl v praxi: 80 % branch coverage. Kritické moduly (platby, bezpečnost): 90+ %.

pytest — moderní testovací framework

pytest je dnes nejpoužívanější Python testovací nástroj. Oproti unittest nepotřebuje třídy, píše se méně kódu.

Základní pytest testy — funkce začínající test_ jsou automaticky nalezeny a spuštěny. Místo assertEqual se používá prostý assert.
# test_kalkulacka.py
import pytest

class Kalkulacka:
    def secti(self, a, b): return a + b
    def vydel(self, a, b):
        if b == 0: raise ZeroDivisionError("Nelze delit nulou!")
        return a / b

# Jednoduchy test - zadna trida potreba
def test_secti():
    calc = Kalkulacka()
    assert calc.secti(2, 3) == 5

def test_vydel_nulou():
    calc = Kalkulacka()
    with pytest.raises(ZeroDivisionError):
        calc.vydel(10, 0)

# Spusteni: pytest test_kalkulacka.py -v
pytest fixture — @pytest.fixture nahrazuje setUp(). Fixture se sdílí mezi testy a může mít různý scope (function, module, session).
import pytest

@pytest.fixture
def kalkulacka():
    """Fixture - vytvoří kalkulačku před každým testem"""
    print("  [setup] vytvarim kalkulacku")
    calc = Kalkulacka()
    yield calc   # testovaci kod bezi tady
    print("  [teardown] uklizim")

def test_secti(kalkulacka):           # fixture se predava jako argument
    assert kalkulacka.secti(10, 5) == 15

def test_odecti(kalkulacka):
    assert kalkulacka.secti(-1, 1) == 0
@pytest.mark.parametrize — elegantní parametrizace bez subTest. Každá kombinace parametrů se zobrazí jako samostatný test v reportu.
import pytest

@pytest.mark.parametrize("a, b, ocekavano", [
    (2,   3,    5),
    (-1,  1,    0),
    (0,   0,    0),
    (100, -50,  50),
])
def test_secti_parametry(a, b, ocekavano):
    calc = Kalkulacka()
    assert calc.secti(a, b) == ocekavano

# pytest test_kalkulacka.py -v
# vypise 4 samostatne testy: test_secti_parametry[2-3-5], [(-1)-1-0], ...
📖 ČÁST 5 — Dokumentace zdrojového kódu

Docstring — dokumentační komentáře

Docstring je string hned za definicí funkce/třídy/modulu. Python ho uloží do __doc__ a nástroje ho čtou pro generování dokumentace.

StylPoužitíPopis
Google styleArgs: / Returns: / Raises:Čitelný, používá Google a mnoho projektů
NumPy styleParameters / Returns / NotesVědecké projekty, pandas, numpy
reStructuredText:param x: / :returns:Sphinx dokumentace, starší styl
Docstring + doctest — doctest umožňuje psát testy přímo do docstringu ve tvaru Python REPL. Spouštíme přes python -m doctest nebo pytest.
def secti(a: float, b: float) -> float:
    """Secte dve cisla a vrati jejich soucet.

    Args:
        a: Prvni scitanec.
        b: Druhy scitanec.

    Returns:
        Soucet a + b jako float.

    Raises:
        TypeError: Pokud a nebo b neni cislo.

    Examples:
        >>> secti(2, 3)
        5
        >>> secti(-1, 1)
        0
        >>> secti(0.1, 0.2)  # doctest: +ELLIPSIS
        0.3...
    """
    if not isinstance(a, (int, float)) or not isinstance(b, (int, float)):
        raise TypeError(f"Ocekavano cislo, dostano: {type(a)}, {type(b)}")
    return a + b

# Spusteni doctestu:
# python -m doctest soubor.py -v
# nebo: pytest --doctest-modules
Type hints a anotace — PEP 484 přidal do Pythonu typové anotace. Neovlivňují běh programu, ale IDE a linters (mypy) je kontrolují. Výrazně zlepšují čitelnost a dokumentují rozhraní.
from typing import Optional, List

def najdi_studenta(studenti: List[dict], id: int) -> Optional[dict]:
    """Najde studenta podle ID.

    Args:
        studenti: Seznam slovniku se studenty.
        id: ID hledaneho studenta.

    Returns:
        Slovnik se studentem nebo None, pokud student neexistuje.
    """
    for s in studenti:
        if s.get("id") == id:
            return s
    return None

# Priklad pouziti:
data = [{"id": 1, "jmeno": "Alice"}, {"id": 2, "jmeno": "Bob"}]
student = najdi_studenta(data, 1)     # Optional[dict] — muze byt None
print(student["jmeno"] if student else "Nenalezen")  # Alice
